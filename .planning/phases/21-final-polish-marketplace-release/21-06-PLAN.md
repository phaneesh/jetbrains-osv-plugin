<title>Audit shadow JAR minimize exclusions and runtime class preservation</title>
<read_first>
  - build.gradle.kts
  - .planning/phases/21-final-polish-marketplace-release/21-RESEARCH.md (section 6.1)
  - .planning/codebase/CONCERNS.md (issue #12)

<action>
In build.gradle.kts, review and update the shadow JAR configuration:

1. Ensure `minimize()` block has explicit excludes for ALL Kotlin stdlib and IntelliJ platform dependencies that are needed at runtime but may be stripped:

```kotlin
minimize {
    exclude(dependency("org.jetbrains.kotlin:.*"))
    exclude(dependency("org.jetbrains:.*"))
    exclude(dependency("com.google.code.gson:.*")) // Gson is explicitly bundled
    exclude(dependency("org.apache.maven:.*"))    // Maven model is explicitly bundled
}
```

2. Add a verification step after `shadowJar` to print the contents for human audit:

```kotlin
tasks.named<com.github.jengelman.gradle.plugins.shadow.tasks.ShadowJar>("shadowJar") {
    // existing config...
    doLast {
        val jarFile = archiveFile.get().asFile
        println("=== Shadow JAR contents ===")
        println("Size: ${jarFile.length() / 1024} KB")
        java.util.jar.JarFile(jarFile).use { jar ->
            jar.entries().asSequence()
                .filter { !it.isDirectory }
                .groupBy { it.name.substringBeforeLast('/') }
                .forEach { (pkg, entries) ->
                    println("  $pkg/: ${entries.size} files")
                }
        }
    }
}
```

3. Verify `./gradlew shadowJar` produces a JAR under 10MB. If larger, investigate unexpected transitive dependencies and add explicit `exclude(dependency("..."))` in the `dependencies` block of the shadow task.

4. After building, run `./gradlew runIde` for a quick smoke test to confirm no `NoClassDefFoundError` at startup.
   </action>
   <acceptance_criteria>

- build.gradle.kts `minimize` block excludes `org.jetbrains.kotlin`, `org.jetbrains`, `com.google.code.gson`, `org.apache.maven`
- build.gradle.kts shadowJar task has a `doLast` block printing JAR size and package file counts
- `./gradlew shadowJar` produces a JAR ≤ 10MB
- `./gradlew runIde` starts without `NoClassDefFoundError` (if runIde is available in environment)
  </acceptance_criteria>
  </task>

In build.gradle.kts, add the `signPlugin` task configuration:

```kotlin
tasks.signPlugin {
    certificateChain.set(System.getenv("SIGN_CERTIFICATE_CHAIN"))
    privateKey.set(System.getenv("SIGN_PRIVATE_KEY"))
    password.set(System.getenv("SIGN_PRIVATE_KEY_PASSWORD"))
}
```

If the `signPlugin` task is not available by default (IntelliJ Gradle Plugin 1.17.0 may not auto-register it), add it explicitly:

```kotlin
import org.jetbrains.intellij.tasks.SignPluginTask

tasks.register<SignPluginTask>("signPlugin") {
    dependsOn("buildPlugin")
    inputArchiveFile.set(tasks.buildPlugin.get().archiveFile)
    certificateChain.set(System.getenv("SIGN_CERTIFICATE_CHAIN"))
    privateKey.set(System.getenv("SIGN_PRIVATE_KEY"))
    password.set(System.getenv("SIGN_PRIVATE_KEY_PASSWORD"))
}
```

If `SignPluginTask` is not available in v1.17.0 (it requires 1.17.3+ or 2.x), document this in the build file as a comment:

```kotlin
// Plugin signing requires IntelliJ Gradle Plugin 1.17.3+ or 2.x.
// For v1.17.0, signing must be done via the JetBrains Marketplace web UI after upload.
// Upgrade to 1.17.4 or 2.x to enable automated signing in CI.
```

Determine which approach applies by checking if `SignPluginTask` can be imported. If not available, leave a detailed comment and defer automated signing to a future build-tool upgrade.
</action>
<acceptance_criteria>

- build.gradle.kts contains EITHER:
  a) a working `signPlugin` task with certificateChain/privateKey/password from env vars, OR
  b) a clear comment block explaining why signing is deferred and what version upgrade is needed
- `./gradlew tasks --all | grep -i sign` shows the signing task (if configured) or clearly documents its absence
  </acceptance_criteria>
  </task>

<read_first>

<action>
Update `.github/workflows/build.yml` with the following changes:

1. Add a matrix strategy to the `build` job for multi-version testing:

```yaml
build:
  name: Build Plugin
  runs-on: ubuntu-latest
  strategy:
    matrix:
      intellij-version: ["2023.3", "2024.1"]
  steps:
    # existing checkout, JDK setup...
    - name: Build plugin with Gradle
      run: ./gradlew buildPlugin -PintellijVersion=${{ matrix.intellij-version }} --no-daemon
```

2. Update the `release` job to produce a properly named ZIP artifact:

```yaml
- name: Build release artifact
  run: |
    VERSION=${GITHUB_REF#refs/tags/v}
    ./gradlew buildPlugin -PpluginVersion=$VERSION --no-daemon
```

3. Add the `publish` job at the end of the workflow:

```yaml
publish:
  name: Publish to JetBrains Marketplace
  runs-on: ubuntu-latest
  needs: [build, test, verify]
  if: startsWith(github.ref, 'refs/tags/v')
  steps:
    - name: Checkout repository
      uses: actions/checkout@v4

    - name: Setup JDK 17
      uses: actions/setup-java@v4
      with:
        java-version: "17"
        distribution: "temurin"
        cache: "gradle"

    - name: Setup Gradle
      uses: gradle/gradle-build-action@v3
      with:
        gradle-version: "8.5"

    - name: Make Gradle wrapper executable
      run: chmod +x ./gradlew

    - name: Build and publish
      env:
        JETBRAINS_TOKEN: ${{ secrets.JETBRAINS_TOKEN }}
      run: |
        VERSION=${GITHUB_REF#refs/tags/v}
        ./gradlew publishPlugin -PpluginVersion=$VERSION -PintellijPublishToken=${{ secrets.JETBRAINS_TOKEN }} --no-daemon
```

4. If `publishPlugin` task is not available (because the IntelliJ Gradle Plugin version doesn't support it), replace the publish step with:

```yaml
- name: Build release artifact (manual upload required)
  run: |
    VERSION=${GITHUB_REF#refs/tags/v}
    ./gradlew buildPlugin -PpluginVersion=$VERSION --no-daemon
- name: Upload artifact for manual marketplace upload
  uses: actions/upload-artifact@v4
  with:
    name: osv-plugin-release
    path: build/distributions/osv-plugin-*.zip
```

</action>

  - The `release` job sets `-PpluginVersion=` from the git tag
  - A `publish` job exists that runs only on version tags (`refs/tags/v`)
  - The publish job uses `secrets.JETBRAINS_TOKEN` or falls back to artifact upload with a clear comment
