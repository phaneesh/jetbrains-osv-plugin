# Plan 21-01: Critical Bug Fixes

## Goal

Fix all CRITICAL technical debt identified in CONCERNS.md before any marketplace release. These are correctness bugs that break core functionality: CacheManager returns fresh instances (no caching), OsVApiService returns fresh instances (broken rate limiting), hardcoded API URL, and System.err pollution. Fixing these is a prerequisite for everything in later waves.

In OsVApiService.kt, modify the call chain to thread the package name through parsing:

1. In `queryVulnerabilities(packageName, ecosystem, version)`, the call to `executeRequest(request)` currently returns parsed results. After receiving the response, pass the `packageName` to parsing. If `parseVulnerabilities` is used (plural, for batch), change its signature to `parseVulnerabilities(responseJson, packageName)`. If a singular `parseVulnerability(vulnObj)` exists, change it to `parseVulnerability(vulnObj, packageName: String)`.

2. In `batchQueryVulnerabilities(dependencies)`, when iterating uncached dependencies and calling parsing for each dep, pass `dep.name` as the package name argument.

3. In the parsing helper(s), populate the Vulnerability `packageName` field. If Vulnerability is a data class, use `vulnerability.copy(packageName = packageName)`. If Vulnerability is a regular class with a mutable `var packageName`, assign it directly.

In Vulnerability.kt, verify that `packageName` is either:

- A `var packageName: String = ""` (mutable with default), OR
- A constructor parameter that can be set via `copy(packageName = ...)`

Ensure no other code breaks from the signature changes. The package name must NOT be parsed from the OSV JSON response (it is not reliably present there); it MUST be threaded from the caller context.
</action>
