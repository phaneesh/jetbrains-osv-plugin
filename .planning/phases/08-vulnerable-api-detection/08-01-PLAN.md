# Phase 8: Vulnerable API Detection

**Date:** 2026-05-07
**Status:** ✅ COMPLETE

## Goal

Detect if vulnerable methods from dependencies are actually called in source code ("reachability analysis"). This mimics Mend.io's premium "reachability" feature using only open data from the OSV database.

## Research

### OSV Database Structure

OSV schema provides `affected[].database_specific.functions` for some vulnerabilities:

```json
{
  "affected": [{
    "database_specific": {
      "functions": [
        "org.apache.logging.log4j.Logger.debug",
        "org.apache.logging.log4j.Logger.error"
      ]
    }
  }]
}
```

Not all vulnerabilities have this data — coverage varies by ecosystem and vuln severity.

### PSI Walking Approach

- Java: `PsiMethodCallExpression` via `PsiRecursiveElementVisitor`
- No full type resolution (too slow, requires compiled classes)
- Method name matching is sufficient for first-pass detection

## Architecture

```
OSV API Response
    ↓ parseVulnerability()
affected[].database_specific.functions
    ↓ parseAffectedFunctions()
List<AffectedFunction>
    ↓ VulnerableApiService.analyzeReachability()
ReadAction.compute → PSI walk source files
    ↓ findVulnerableCallSites()
List<VulnerableCallSite>
    ↓ ReachabilityResult
reachable = callSites.isNotEmpty()
```

## Key Decisions

- **No type resolution**: Matching by method name only. Full FQN resolution requires compiled classes and is slow; method name is the 80% solution.
- **Heuristic fallback**: When `database_specific.functions` is absent, infer from summary text (e.g. "Logger.debug() is vulnerable").
- **Source file walking**: Uses `VfsUtilCore.visitChildrenRecursively` + file extension filtering instead of `FileTypeIndex` (which requires compiled class availability for file type instances).
- **ReadAction required**: All PSI access wrapped in `ReadAction.compute` per IntelliJ threading model.

## Limitations / TODOs

1. **No Kotlin call expression handling**: Currently only Java `PsiMethodCallExpression`. Kotlin PSI (`KtCallExpression`) would require reflection or Kotlin plugin dependency.
2. **No type resolution**: Cannot distinguish `logger.debug()` vs `other.debug()` if both have method name `debug`.
3. **Limited OSV function coverage**: Only ~30% of OSV entries have `database_specific.functions`.
4. **Performance on large codebases**: PSI walking all source files is O(n) where n = total source lines. Should cache per project and use incremental updates.
