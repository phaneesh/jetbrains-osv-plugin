# Phase 10: Basic SAST / Taint Analysis

**Date:** 2026-05-07
**Status:** ✅ COMPLETE

## Goal

Implement lightweight static application security testing (SAST) for common vulnerability classes. Snyk, Qodana, and Mend.io all offer SAST; implementing a basic version positions the OSV plugin as a comprehensive security suite.

## Implementation

### Architecture Overview

```
Project Source Files (.java, .kt, .groovy)
    ↓ SastAnalyzer.analyzeProject()
ReadAction.compute + VfsUtilCore.visitChildrenRecursively
    ↓ Per-file detectors
  SqlInjectionDetector
  PathTraversalDetector
  XssDetector
    ↓ PsiRecursiveElementVisitor
    ↓ Pattern matching on PsiMethodCallExpression
List<SastFinding>
```

### Detectors

#### 1. SqlInjectionDetector (`SqlInjectionDetector.kt`)
- Detects: `Statement.executeQuery()`, `executeUpdate()`, `execute()`, `Connection.prepareStatement()`
- Flags: Arguments containing user input patterns (`getParameter`, `getQueryString`, `getHeader`, etc.)
- Severity: **CRITICAL**
- Remediation: "Use PreparedStatement with parameterized queries"

#### 2. PathTraversalDetector (`PathTraversalDetector.kt`)
- Detects: `new FileInputStream(...)`, `new File(...)`, `Paths.get(...)`, `Files.readAllBytes(...)`
- Flags: Paths constructed with untrusted input
- Severity: **HIGH**
- Remediation: "Validate and sanitize user-provided paths. Use canonical paths."

#### 3. XssDetector (`XssDetector.kt`)
- Detects: `PrintWriter.print()`, `println()`, `write()`, `Model.addAttribute()`
- Flags: Untrusted input flowing into HTML/response output
- Severity: **HIGH**
- Remediation: "Use a templating engine with auto-escaping or HTML-encode user input"

### Files Created

1. **`src/main/java/io/dyuti/osvplugin/sast/SastAnalyzer.kt`** (~100 lines):
   - `analyzeProject(project)`: Entry point, orchestrates all detectors
   - `collectJavaSourceFiles()`: VFS-based file collection using `VirtualFileVisitor`

2. **`src/main/java/io/dyuti/osvplugin/sast/SastFinding.kt`** (data class):
   - `ruleId`: Rule identifier (SQL-INJECTION, PATH-TRAVERSAL, XSS-REFLECTED)
   - `severity`: SastSeverity enum (CRITICAL, HIGH, MEDIUM, LOW, INFO)
   - `message`, `filePath`, `lineNumber`, `remediation`

3. **`src/main/java/io/dyuti/osvplugin/sast/SqlInjectionDetector.kt`** (~150 lines):
   - PSI-based method call detection
   - `isConstantExpression()`: Distinguishes compile-time literals from dynamic expressions
   - `containsUserInput()`: Recursive visitor for untrusted input patterns

4. **`src/main/java/io/dyuti/osvplugin/sast/PathTraversalDetector.kt`** (~130 lines):
   - Detects both constructor calls (`new File(...)`) and static method calls (`Paths.get(...)`)
   - Reflection-based `PsiNewExpression` handling (class reference + argument list)

5. **`src/main/java/io/dyuti/osvplugin/sast/XssDetector.kt`** (~120 lines):
   - Detects response writers (`PrintWriter.println`, `write`) and Spring MVC (`Model.addAttribute`)
   - User input pattern matching via recursive visitor

6. **`src/test/java/io/dyuti/osvplugin/sast/SastAnalyzerTest.kt`** (6 tests):
   - Model field verification
   - Severity enum completeness
   - Instance creation for all 3 detectors
   - Finding equality

### Design Decisions

- **Pattern-matching over full data-flow**: Full taint analysis requires a constraint solver (like CodeQL, Semgrep). We use lightweight PSI pattern matching for speed and simplicity.
- **No inter-procedural analysis**: Each detector only looks within a single file. Cross-file taint tracking would require a call graph.
- **Keyword-based user input detection**: Looks for known dangerous APIs (`request.getParameter`, `HttpServletRequest` methods) rather than type inference.
- **ReadAction.compute**: All PSI access happens under IntelliJ's read lock.

## Limitations

| Limitation | Explanation |
|------------|-------------|
| No full data-flow tracking | Cannot trace `String x = request.getParameter("x"); stmt.executeQuery("SELECT ... " + x);` |
| No inter-procedural taint | If user input travels through 2+ methods, it's not detected |
| False positives | A parameterized query built with `String.format()` might be flagged incorrectly |
| Java only | Kotlin and Groovy detection not yet implemented |
| No stored XSS/DOM XSS | Requires database/JS analysis respectively |
